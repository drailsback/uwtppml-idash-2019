## Overview

Source code for the iDASH 2019 Track IV competition. 

Publication: [link](https://link.springer.com/article/10.1186/s12920-020-00869-9)

## Prerequisites

* Install [rust](https://rust-lang.org/tools/install/)
* To replicate iDASH 2019 results, use the [BC-TCGA](https://www.kaggle.com/datasets/orvile/gene-expression-profiles-of-breast-cancer), [GSE2034](https://www.kaggle.com/datasets/saurabhshahane/gene-expression-profiles-of-breast-cancer) data sets, which are publicly available. 

## Setup Instructions

1. **Configure TI_Settings** with the number of shares of each type needed for the protocol.
2. **Configure Party0_Settings** with paths to its input/output files.
3. **Configure Party1_Settings** with paths to its input/output files.
   
   **Configuration Fields Details:**
   * `x_input_path` *(input)*: Location of secret shares of the data set (`.csv`).
   * `y_input_path` *(input)*: Location of secret shares of the class vector (`.csv`).
   * `xor_shares_path` *(output)*: Location to store binary shares sent from the Trusted Initializer (TI) during execution (`.csv`).
   * `additive_shares_path` *(output)*: Location to store additive shares sent from the TI during execution (`.csv`).
   * `attribute_count` *(input)*: Number of columns in data set `x`.
   * `instance_count` *(input)*: Number of rows in data set `x`.
   * `learning_rate` *(input)*: Learning rate to use for the Logistic Regression (LR) protocol.
   * `decimal_precision` *(input)*: Number of bits reserved for fixed-point decimal values (must be consistent with the way `x` and `y` are shared).
   * `integer_precision` *(input)*: Number of bits needed to represent max/min integer values during the protocol. 
     * *Note:* `decimal_precision + integer_precision < 32` (operations are performed on a ring mod $2^{64}$).
   * `output_path` *(output)*: Location to store the final weights generated by the model in the clear (`.csv`).

   > ⚠️ **Important Configuration Notes:**
   > * `x` and `y` must be separate files representing correlated inputs pre-generated by a client.
   > * `xor_share_path` and `additive_shares_path` **MUST** be different for each party.
   > * `output_path` **MAY** be different, but the generated file will contain identical weights for both parties.
   > * All other settings fields **MUST** match exactly between parties.

4. Navigate to `idash2019_rust` and build the project:
   ```bash
   cargo build

```

5. Open a terminal and launch the Trusted Initializer (**must be running before the parties**):
```bash
cargo run TI_Settings.toml

```


6. In a separate terminal, start Party 0:
```bash
cargo run Party0_Settings.toml

```


7. In a third terminal, start Party 1:
```bash
cargo run Party1_Settings.toml

```


8. After execution finishes, the weights file will be generated.
9. Edit the paths in `accuracy_test.py` to point to the newly generated weights location and your `x`/`y` test files.
10. Run the accuracy test using:
```bash
python accuracy_test.py <fold number>

```
